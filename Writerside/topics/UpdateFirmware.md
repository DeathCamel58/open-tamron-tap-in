## Firmware decryption

This project decrypts Tamron firmware packages in-memory using a very small XOR stream cipher and a per-file decryption key. The reference implementation is in `src/types/DecryptionFirmware.ts`.

### High-level flow

1. Build a 32-byte base key from the ASCII passphrase `EncryptionToolBaseKey` by repeating/truncating it to 32 bytes.
2. Accept a firmware decryption key as a 64-character hex string (32 bytes). If it is not exactly 64 hex chars, decryption fails.
3. Derive the working 32-byte stream key by XOR-decrypting the 32-byte hex key with the base key (apply the XOR stream once across those 32 bytes). This yields the final per-file decryption key.
4. Initialize the stream cipher with the derived key and start at index 0. The stream position advances continuously as bytes are processed; it is not reset between header and data within a block.
5. Process the firmware as a sequence of blocks, each consisting of a 1024-byte encrypted header followed by an encrypted data region whose size is found inside the decrypted header. For each block:
   - Decrypt the 1024-byte header using the XOR stream.
   - Read `dataSize` from the header at offset 64 (little-endian 32-bit).
   - Read the expected CRC from the header at offset 70 (2 bytes, little-endian).
   - Decrypt `dataSize` bytes of payload with the same stream (continuing the key index).
   - Compute a 16-bit CRC over the decrypted payload and compare to the headerâ€™s CRC. If it does not match, abort.
   - Append the decrypted header and decrypted data to the output.
6. Repeat until the end of the input.

### Header layout and limits

- Header length per block: 1024 bytes
- `dataSize` field: offset 64, 4 bytes, little-endian unsigned
- `dataCrc` field: offset 70, 2 bytes, little-endian
- Maximum allowed `dataSize` per block: 1,048,576 bytes (`0x100000`). Larger values are treated as invalid file format.

### XOR stream cipher details

- The cipher uses a 32-byte key and performs `out[i] = in[i] XOR key[keyIndex]`, with `keyIndex` incrementing modulo 32.
- The same routine is used both to derive the working key (by XOR-ing the provided 32-byte key with the base key) and to decrypt firmware bytes.
- For a given decryption run, the stream index starts at 0 once and advances across the entire file, so the header and its following data share a continuous keystream.

### CRC over decrypted data

- A 16-bit CRC is computed over the decrypted payload bytes of each block and must match the value stored in the header.
- Implementation uses a precomputed 256-entry table with polynomial 0x1021 and an initial value of 0 (no final XOR), returning the result as two bytes in little-endian order.

## Firmware Update Process

1. Decrypt the firmware
2. Firmware update command packet sent: `0F 0B 00 00 03 00 FD 00 00 2A 01 F0`
3. Data RX'd from lens (this isn't a structured frame, just one byte): `0x43`
4. Push decrypted firmware file
  1. Firmware data chunk sent with structure: `0x02 <INCREMENTING> <DECREMENTING> <FIRMWARE_CHUNK> <TWO_BYTE_CHECKSUM>` (examples in [](#firmware-update-chunks))
   - `<INCREMENTING>` is a byte that increments starting at `0x01`
   - `<DECREMENTING>` is a byte that decrements starting at `0xfe`
   - `<FIRMWARE_CHUNK>` is the bytes of the firmware (seems to vary)
   - `<TWO_BYTE_CHECKSUM>` is a checksum generated the same way as standard data frames.<br />**NOTE:** The CRC is **only** computed over `<FIRMWARE_CHUNK>`
  2. RX `0x06` from the lens. This likely is a data received message
5. Send `0x04`
6. RX `0x15`
7. Send `0x04`
8. RX `0x15`
9. Send [](RequestLensDisconnect.md)
10. Send [](RequestLensConnect.md)
